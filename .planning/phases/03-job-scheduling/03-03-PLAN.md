## Objective

Complete the job scheduling system with error recovery (exponential backoff
retries), failure logging to rotating files, and database freshness tracking.
Implement monitoring endpoint to verify data freshness.

**Purpose**: Enable operators to monitor scraper health, debug failures through
logs, and confirm data stays fresh within 24-hour SLA.

**Output**:

- ScraperMetadata table in database (last_successful_run timestamps)
- Retry logic in both job functions (max 3 attempts, 1-hour window, exponential
  backoff)
- Logger configured for rotating failure log files
- /health/data-freshness endpoint showing freshness status per scraper
- Both job functions updated to track metadata on success

       - Wrap job in retry wrapper: async function deputiesJobWithRetry(attempt = 1)

         * Try to run runPerson()
         * On success: call updateScraperMetadata('deputies', true), log success with timestamp and record count, return success result
         * On failure: log error with timestamp and attempt count

         * Calculate exponential backoff: 30s for 1st retry, 60s for 2nd, 120s for 3rd
         * Wait before retry
         * Recursively call deputiesJobWithRetry(attempt + 1)
       - If attempt === 3 OR 1 hour elapsed:
         * Call updateScraperMetadata('deputies', false, error.message)
         * Log final failure, do NOT retry again
         * Return failure result
       - Export default as deputiesJob (calls deputiesJobWithRetry(1))
    2. Update apps/ingestion/src/jobs/voting.ts with identical retry pattern, but:
       - Call runVoting() instead of runPerson()
       - Call updateScraperMetadata('voting', ...) instead of 'deputies'
  </action>

  </verify>
  <done>
    - deputies.ts has retry logic: max 3 attempts, exponential backoff (30s, 60s, 120s)
    - voting.ts has identical retry logic with voting-specific calls

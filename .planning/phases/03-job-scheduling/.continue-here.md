## Decisions Made

**Phase 3 architectural choices (from CONTEXT.md, all locked):**

1. **Bree for scheduling** - Built-in worker thread isolation, cron support,
   good documentation
2. **Single process + try-catch** - Simple implementation, sufficient isolation,
   reduces resource overhead
3. **Max 3 retries within 1-hour window** - Catches transient failures without
   hammering external sites
4. **Failure logs to files only** - Lightweight, no database overhead for every
   run
5. **Fixed UTC scheduling** - 2 AM (deputies), 3 AM (voting) prevents concurrent
   DB writes
6. **Database tracking** - last_successful_run timestamp per scraper for
   freshness verification

**Session-specific decisions:**

- Imported `runPersonStandalone()` and `runVotingStandalone()` instead of using
  module-level browser variable (enables job file imports)
- Used exponential backoff formula: 30s (1st retry), 60s (2nd retry), 120s (3rd
  retry)

## Context / Mental Model

**Phase 3 workflow:**

1. Scheduler (Bree) starts on app launch
2. Scheduler registers jobs from job registry (deputies.ts, voting.ts)
3. Each job configured with cron expression runs at scheduled time
4. Job execution wrapped in try-catch → error doesn't cascade
5. Job calls scraper function (runPersonStandalone/runVotingStandalone)
6. On success: update database metadata (lastSuccessfulRun timestamp), no
   logging
7. On failure: log to rotating failure file, retry up to 3 times with
   exponential backoff within 1 hour
8. Monitoring endpoint (/health/data-freshness) queries metadata to show
   freshness status

**Key insight:** Retry window prevents retry cascade during systemic failures.
Within 1 hour, up to 3 attempts catch transient issues. After 1 hour or 3
failures, wait until next scheduled run (24h later).
